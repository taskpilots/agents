# 互联网分析 Agent 系统设计方案

## 本地启动最小步骤

当前仓库默认以 `InMemory` 模式运行，适合本地初始化与联调。`Postgres`、`pgvector`、`Docker Compose` 和数据库脚本属于后续基础设施阶段，本轮不要求先接入。

### 1. 启动后端

```bash
dotnet run --project TaskPilots.TaskAgents.WebApis
```

默认开发地址沿用 `TaskPilots.TaskAgents.WebApis/Properties/launchSettings.json` 中的 `http://localhost:5131`。

### 2. 启动前端开发模式

```bash
pnpm --dir taskagentsweb install
pnpm --dir taskagentsweb dev
```

前端开发服务器会把 `/api/*` 和 `/agent-hub` 代理到 `http://localhost:5131`，因此不需要额外修改前端接口地址。

### 3. 构建并发布前端到后端静态目录

```bash
pnpm --dir taskagentsweb build
```

构建完成后，前端产物会自动复制到 `TaskPilots.TaskAgents.WebApis/wwwroot`，后端可直接静态托管管理控制台。

### 4. 验证

```bash
dotnet test TaskPilots.TaskAgents.slnx
pnpm --dir taskagentsweb test
```

## 1. 项目概述

本项目目标是构建一个**独立部署、长期运行、可审计、可审批、半自主执行**的互联网分析 Agent 系统。

系统不是单次对话式助手，而是一个能够持续接收输入、管理任务生命周期、执行多阶段分析、在必要时请求审批并输出结果的运行平台。

系统采用如下核心模式：

- **单体应用 + 内置 Worker**
- **Main Agent + SubAgent**
- **状态机 + 调度器 + 工具层**
- **完整日志、Checkpoint、Execution Path**
- **审批与通知内建**

系统能够处理以下输入来源：

- 定时任务
- 人工创建任务
- 外部 API / Webhook
- 邮箱来信

系统能够对任务进行规划、拆解、执行、总结，并在受控权限范围内完成通知、邮件回复与结果输出。

## 2. 技术选型

本项目采用以下技术方案：

### 后端
- **.NET 10**

用于构建核心服务、任务管理、调度、状态机、Agent Runtime、工具层、审批流、通知与邮箱处理能力。

### 前端
- **React + Vite**

用于构建管理控制台，包括任务列表、运行记录、审批界面、邮件归档查看、日志检索与系统配置页面。

### 数据库
- **PostgreSQL**

作为主业务数据库，用于保存任务、事件、运行记录、审批记录、邮件归档、日志索引、配置等核心数据。

### 向量能力
- **pgvector**

用于存储向量数据，支持任务上下文检索、知识片段召回、历史结论辅助参考等增强能力。

### 文件存储
- **本地磁盘**

用于保存附件、解析产物、报告导出文件、原始归档邮件、日志快照等文件型数据。

### 模型能力
- **OpenAI Responses API**

用于 Main Agent 与 SubAgent 的理解、规划、总结、结构化输出、回复草稿生成等智能能力。

### 部署方式
- **Docker Compose**

用于本地开发、测试环境及中小规模独立部署场景下的统一启动与服务编排。

## 3. 系统定位

本系统本质上是一个：

> **长期运行的互联网分析 Agent 平台**

它具备如下特点：

1. 能接收多种输入源
2. 能以任务为中心进行生命周期管理
3. 能以 Main Agent 统筹规划，以 SubAgent 执行具体工作
4. 能通过状态机确保流程可控
5. 能通过调度器长期运行
6. 能在关键动作前引入审批
7. 能完整记录执行日志、检查点与执行路径
8. 能通过邮箱完成外部输入与输出闭环

## 4. 架构原则

### 4.1 系统控制流程，模型负责智能判断

LLM 不直接掌控系统流程。  
系统负责：

- 状态管理
- 任务编排
- 调度执行
- 审批控制
- 日志记录
- 文件归档
- 权限边界
- 邮件收发与事件化

模型负责：

- 目标理解
- 内容分析
- 任务拆解建议
- 总结汇报
- 回复草稿
- 决策建议输出

### 4.2 显式状态驱动

任务、运行、审批、邮件处理均采用显式状态推进，避免“隐式对话流”失控。

### 4.3 默认可审计

所有关键动作均保留记录，包括：

- 输入来源
- 决策建议
- 工具调用
- 状态变更
- 审批结果
- 对外通知
- 邮件往来
- 执行路径与Checkpoint

### 4.4 默认最小权限

敏感动作默认受限，特别是：

- 对外发信
- 外部写操作
- 高影响任务变更
- 未经确认的信息发布

## 5. 总体架构

系统采用**单体应用 + 内置 Worker** 模式，统一承载：

- API 服务
- 管理后台接口
- 任务编排
- 后台执行器
- 调度器
- 邮箱处理
- 审批流
- 通知能力

整体逻辑结构如下：

```text
[输入层]
  ├─ Scheduler
  ├─ Manual Task API
  ├─ Webhook/API Input
  └─ Mailbox Input

        ↓

[接入与事件层]
  ├─ Event Ingestion
  ├─ Mailbox Processor
  ├─ Request Parser
  └─ Event Store

        ↓

[任务与流程控制层]
  ├─ Task Manager
  ├─ State Machine
  ├─ Run Orchestrator
  ├─ Approval Manager
  └─ Notification Manager

        ↓

[Agent Runtime]
  ├─ Main Agent
  └─ SubAgent Runtime

        ↓

[工具层]
  ├─ Web Search
  ├─ Browser / Extractor
  ├─ File Processor
  ├─ Email Sender
  └─ Knowledge Retrieval

        ↓

[数据与归档层]
  ├─ PostgreSQL
  ├─ pgvector
  └─ Local File Storage
```

## 6. 架构模式说明

## 6.1 单体应用 + 内置 Worker

系统采用单体模式，避免早期过度拆分。  
在一个应用中同时包含：

- 对外 API
- 管理后台接口
- 后台任务消费
- 调度执行
- 邮件轮询
- 审批处理
- 通知分发

这样做的优点：

- 结构简单
- 便于开发与部署
- 状态管理集中
- 审计链路完整
- 适合当前阶段的独立部署需求

系统内部通过模块边界划分职责，而不是通过微服务拆分。

---

## 6.2 Main Agent + SubAgent

系统采用双层 Agent 结构：

### Main Agent
负责：

- 理解输入目标
- 识别任务类型
- 制定计划
- 拆解子任务
- 汇总阶段成果
- 判断是否需要审批
- 生成回复或汇报建议

### SubAgent
负责：

- 执行具体分析任务
- 调用限定范围内工具
- 返回结构化结果
- 不承担全局流程控制责任

这种结构的意义在于：

- 将高层规划和低层执行分离
- 控制上下文规模
- 支持并行分析
- 提高任务执行清晰度和可审计性

---

## 6.3 状态机 + 调度器 + 工具层

这是系统长期稳定运行的核心。

### 状态机
用于控制任务、运行和审批的流转。

### 调度器
用于处理：

- 定时任务触发
- 延迟重试
- 周期巡检
- 邮箱轮询
- 超时检查
- 审批超时处理

### 工具层
所有外部能力通过工具层暴露，统一进行：

- 权限控制
- 输入校验
- 调用记录
- 失败处理
- 风险约束

---

## 7. 核心模块设计

## 7.1 Event Ingestion

负责统一接收系统输入，并将其转为内部事件。

输入包括：

- 定时触发
- 手工创建任务
- Webhook / API
- 邮箱来信

它的目标不是直接执行业务逻辑，而是将输入标准化，交由后续流程处理。

---

## 7.2 Task Manager

负责任务对象的创建、更新、归档与关联管理。

主要职责：

- 创建任务
- 更新任务状态
- 维护任务上下文
- 关联邮件线程、附件、审批记录、执行结果
- 提供任务查询能力

任务是系统中的核心业务实体。

---

## 7.3 State Machine

负责显式控制任务推进过程，防止执行过程失控或状态不一致。

适用对象包括：

- Task
- Agent Run
- Approval
- Mail Processing

状态机的存在，使系统具备：

- 可恢复性
- 可解释性
- 可审计性
- 可干预性

---

## 7.4 Run Orchestrator

负责驱动一次任务或子任务的实际执行过程。

职责包括：

- 读取待执行任务
- 调用 Main Agent 或 SubAgent
- 控制步骤推进
- 记录执行路径
- 写入 Checkpoint
- 将结果交回状态机和任务管理模块

它是系统的执行枢纽。

---

## 7.5 Main Agent Runtime

负责为 Main Agent 提供运行环境。

输入可能包括：

- 当前事件
- 任务描述
- 历史上下文
- 邮件内容
- 附件摘要
- 相关知识召回结果
- 当前状态与约束条件

输出为结构化决策建议，例如：

- 创建任务
- 调整优先级
- 生成子任务
- 等待更多输入
- 请求审批
- 生成回复草稿

---

## 7.6 SubAgent Runtime

负责承载具体执行型 Agent。

主要用于：

- 搜索与浏览网页
- 分析资料
- 提取事实
- 分类整理
- 形成局部结论

SubAgent 生命周期较短、目标较明确、工具权限较窄。

---

## 7.7 Mailbox Processor

邮箱模块是系统中的标准输入组件，不是附属功能。

其职责包括：

- 定期轮询邮箱
- 拉取新邮件
- 去重
- 解析正文与附件
- 归档原始邮件
- 判断是否与已有任务或审批相关
- 生成内部事件

邮箱来信可能表示：

- 新分析请求
- 任务补充信息
- 审批回复
- 外部反馈
- 附件资料提交
- 普通通知

---

## 7.8 Approval Manager

负责敏感动作审批。

典型审批场景：

- 对外发送正式邮件
- 发布外部结论
- 调用高影响外部动作
- 对高优先级任务做重大变更

审批通过后继续执行，审批拒绝后终止、回退或重新规划。

---

## 7.9 Notification Manager

负责对系统内外的消息通知进行统一处理。

通知形式可包括：

- 站内提醒
- 审批通知
- 任务状态通知
- 邮件通知

通知与审批分离，但可以相互协同。

---

## 7.10 Tool Layer

工具层负责封装外部操作能力，并将其置于系统控制之下。

典型工具包括：

- Web Search
- Browser / 内容提取
- File Processor
- Email Sender
- Knowledge Retrieval

工具层只提供能力，不持有业务决策权。

---

## 8. 数据与存储设计

## 8.1 PostgreSQL

PostgreSQL 作为主数据库，统一保存：

- 任务
- 事件
- 运行记录
- 审批记录
- 邮件归档元数据
- 通知记录
- 配置
- 执行路径索引
- Checkpoint 元数据

它是系统事实来源的核心。

---

## 8.2 pgvector

用于支持向量检索能力，例如：

- 历史任务上下文召回
- 相关邮件片段召回
- 报告片段查找
- 知识增强检索

pgvector 作为 PostgreSQL 的扩展能力存在，避免引入额外独立向量数据库。

---

## 8.3 本地磁盘

本地磁盘用于保存文件型数据，例如：

- 邮件原文
- 附件
- 报告导出
- 中间产物
- 原始抓取文件
- 日志快照
- Checkpoint 对应的大对象内容

适合当前阶段独立部署场景。

---

## 9. 模型能力设计

系统采用 **OpenAI Responses API** 作为统一模型接口。

模型主要用于：

- Main Agent 规划
- SubAgent 分析
- 邮件内容理解
- 附件内容摘要
- 结构化结果生成
- 报告与回复草稿生成

模型输出应尽量采用结构化形式，以便系统做后续控制和校验。

模型不是系统调度中心，而是受控智能组件。

---

## 10. 任务驱动模型

## 10.1 任务是核心对象

无论输入来自：

- 定时器
- 用户手工创建
- Webhook
- 邮箱

最终都应尽量落到任务模型上进行统一管理。

---

## 10.2 任务生命周期

任务从输入到结束，通常经过：

```text
created
→ triaged
→ planned
→ waiting_for_execution
→ running
→ waiting_for_approval / waiting_for_external_input
→ partially_completed / completed
→ archived
```

任务状态由系统推进，不由模型自由修改。

---

## 10.3 子任务与运行单元

每个任务可以包含多个子任务或执行单元，用于承载：

- 网页搜索
- 多源资料阅读
- 附件处理
- 汇总草稿生成

这些执行单元既能独立记录日志，也能作为整体执行路径的一部分。

---

## 11. Execution Path 与 Checkpoint

这是本系统的重要特征之一。

## 11.1 Execution Path

Execution Path 用于记录任务执行过程中的完整轨迹，包括：

- 收到什么输入
- 做了什么判断
- 创建了哪些子任务
- 调用了哪些工具
- 产生了哪些中间结果
- 在哪里等待审批
- 最终如何结束

它提供强可追溯性，是调试、审计和复盘的重要基础。

---

## 11.2 Checkpoint

Checkpoint 用于记录关键执行阶段的可恢复快照。

其目的包括：

- 程序异常后恢复执行
- 长任务分段续跑
- 审批后继续从既有阶段推进
- 复现实验和问题排查

Checkpoint 不意味着完整复制所有上下文，而是保留足够恢复当前阶段的关键状态。

---

## 12. 日志与审计设计

系统要求具备完整日志能力。

日志范围包括：

- 输入接入日志
- 任务状态变更日志
- Agent 决策日志
- SubAgent 执行日志
- 工具调用日志
- 邮件处理日志
- 审批日志
- 通知日志
- 错误日志
- Checkpoint 与恢复日志

审计视角下，应可追踪：

- 谁发起了任务
- 任务基于什么输入产生
- Agent 做过哪些判断
- 哪些动作是自动执行的
- 哪些动作经过审批
- 对外发送过哪些内容
- 最终结果是什么

---

## 13. 审批与通知

## 13.1 审批内建

审批不是外围模块，而是系统内建能力。

审批主要用于控制高风险动作，避免 Agent 在无确认情况下直接执行高影响行为。

审批流应与任务状态流结合，使任务可以自然进入：

- 等待审批
- 审批通过后继续
- 审批拒绝后重规划或终止

---

## 13.2 通知内建

通知系统用于：

- 告知审批人
- 告知任务状态变化
- 推送异常提醒
- 发送阶段性结果
- 对外回信

通知可以作为执行结果的输出层，但是否真正外发，受审批与策略控制。

---

## 14. 邮箱闭环设计

邮箱在本系统中既是输入通道，也是输出通道。

### 输入侧
邮箱来信可转为：

- 新任务
- 任务补充
- 审批回复
- 外部反馈
- 附件资料

### 输出侧
系统可发送：

- 分析结果
- 阶段性报告
- 审批通知
- 回复草稿确认后的正式邮件

邮箱与任务、审批、日志需要形成完整关联，保证消息链条可回溯。

---

## 15. 前端控制台定位

前端采用 **React + Vite**，主要承担管理与运营能力，而不是 Agent 本身。

控制台建议覆盖以下视图：

- 任务列表与详情
- 执行记录查看
- 审批处理
- 邮件归档查看
- 日志与执行路径查看
- 通知中心
- 任务手工创建与干预
- 系统配置与策略管理

前端的核心目标是：

- 提供透明度
- 提供可干预能力
- 提供审计与运维视角

---

## 16. 部署方式

系统采用 **Docker Compose** 部署。

部署对象主要包括：

- 主应用容器
- PostgreSQL
- 必要的辅助服务

该模式适合：

- 单组织独立部署
- 本地开发与测试
- 中小规模生产运行

在当前阶段，这种部署方式能兼顾：

- 简单性
- 可维护性
- 可复制性

---

## 17. MVP 范围建议

第一阶段建议聚焦于打通最关键闭环。

### MVP 必做能力

1. 单体应用基础框架
2. Task / Event / Run 核心模型
3. Main Agent + SubAgent 基础运行机制
4. 状态机与基础调度器
5. 邮箱接入与解析
6. Web Search / Browser / File Processor 基础工具
7. 审批机制基础版
8. 通知与邮件发送
9. 完整日志、Execution Path、Checkpoint 基础能力
10. 基础管理控制台

---

## 18. 后续增强方向

在 MVP 稳定后，可以逐步增强：

- 更复杂的工作流能力
- 更精细的权限控制
- 更强的知识召回能力
- 更丰富的附件解析
- 多邮箱支持
- 更高级的人工协作与干预机制
- 更强的运行监控与告警能力

---

## 19. 非目标

当前阶段不作为重点：

- 微服务化拆分
- 大规模分布式调度
- 通用型企业自动化平台
- 无审批的完全自治 Agent
- 复杂多租户体系
- 过度复杂的插件生态

---

## 20. 最终定义

本系统可以定义为：

> **一个基于 .NET 10 构建、采用单体应用加内置 Worker 架构的长期运行 Agent 平台。系统以 Main Agent 负责规划和总结，以 SubAgent 负责具体执行，以状态机、调度器和工具层保障流程受控；以 PostgreSQL、pgvector 与本地文件存储承载任务、上下文与归档；以 OpenAI Responses API 提供智能能力；并内建日志、Checkpoint、Execution Path、审批与通知机制，实现对互联网分析任务的持续、可审计、可审批的半自主执行。**
